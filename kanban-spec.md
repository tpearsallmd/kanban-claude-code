# Local Kanban Board — Spec & Design

## Overview
A single-repo kanban board for managing development tasks collaboratively between a human and Claude Code. The UI (`kanban.html`) and data (`kanban.json`) live in the same directory. Claude Code reads and writes the JSON directly; the human uses the browser UI for planning and prioritization. Served via VS Code Live Server for real-time file read/write.

---

## Files

```
/kanban
├── kanban.html       # The entire UI — single file, no dependencies
├── kanban.json       # The data file — shared between UI and Claude Code
├── serve.js          # Tiny Node.js HTTP server (read + write on localhost:5555)
└── kanban-spec.md    # This file
```

## Columns & Workflow

```
Backlog → Ready → Design → In Progress → Testing → Review → Done
```

| Column | Owner | Purpose |
|---|---|---|
| **Backlog** | Human | All ideas, bugs, enhancements — unsorted |
| **Ready** | Human/Claude | Groomed, prioritized, enough detail to act on |
| **Design** | Claude (Designer) | Plan approach, update card description with design, then move to Ready for Dev |
| **In Progress** | Claude (Developer) | Actively writing code |
| **Testing** | Claude (Tester) | Write/run tests, verify implementation, file new cards for issues found |
| **Review** | Human | Human reviews Claude's work — approve, request changes, or kick back |
| **Done** | — | Merged/complete |

### Column Entry Policies

| Column | Entry Criteria |
|---|---|
| **Ready** | Description has acceptance criteria, relevant file paths, and edge cases. Card `requirements` section populated. Card is small enough for one session. |
| **Design** | Claude has read the card and confirmed it's actionable. `design` section updated with approach, affected files, and risks. |
| **In Progress** | Design is complete (or card is simple enough to skip). Only move here when actively coding. |
| **Testing** | Code is written and committed. `implementationNotes` updated with what changed. `testPlan` updated with how to test. **Relevant docs in `docs/` folder are updated** (architecture, API, deployment, etc.). |
| **Review** | Tests pass. Docs updated. Human reviews the diff — approves, requests changes, or kicks back. |
| **Done** | Human approved. Code merged or confirmed working. |

### Claude's Readiness Gate

Before moving a card from **Ready** to **Design** or **In Progress**, Claude must:

1. Read the card description and assess whether it has enough context to act on
2. If unclear, update the description with clarifying questions and leave it in Ready
3. If the card is too large, recommend breaking it into multiple cards in Backlog
4. Only move the card forward when the description is actionable

## kanban.json Schema

```json
{
  "version": 1,
  "repo": "SI_RDM",
  "columns": ["Backlog", "Ready", "Design", "In Progress", "Testing", "Review", "Done"],
  "wipLimits": {
    "Design": 2,
    "In Progress": 3,
    "Testing": 3,
    "Review": 5
  },
  "cards": [
    {
      "id": "card_1741234567_abc",
      "title": "Add auth middleware",
      "description": "Implement JWT auth middleware for all protected routes",
      "type": "enhancement",
      "priority": "high",
      "column": "Ready",
      "created": "2026-03-12T10:00:00Z",
      "updated": "2026-03-12T11:00:00Z",
      "size": "M",
      "blocked": false,
      "blockedReason": "",
      "tags": ["backend", "security"],
      "requirements": "Must support JWT and session tokens...",
      "design": "Add middleware at router level, check Authorization header...",
      "implementationNotes": "",
      "testPlan": "",
      "reviewNotes": ""
    }
  ]
}
```

### Field Definitions
- **id** — unique string, format `card_{timestamp}_{random3}`, generated by UI or Claude
- **title** — short task name, required
- **description** — longer context; Claude uses this for session bootstrapping. Should include acceptance criteria, relevant file paths, and edge cases when available.
- **type** — `enhancement` or `defect`
- **priority** — `high`, `medium`, or `low`
- **size** — t-shirt size estimate: `XS`, `S`, `M`, `L`, or `XL`
- **blocked** — boolean, true if the card is blocked on something external
- **blockedReason** — string explaining what the card is blocked on (only when blocked is true)
- **column** — must match one of the values in the top-level `columns` array
- **created / updated** — ISO 8601 timestamps
- **tags** — optional array of strings
- **requirements** — optional string; acceptance criteria, constraints, edge cases (populated during Ready/Design)
- **design** — optional string; approach, affected files, architecture notes (populated during Design)
- **implementationNotes** — optional string; what was changed, key decisions (populated during In Progress)
- **testPlan** — optional string; how to test, expected results, test commands (populated during Testing)
- **reviewNotes** — optional string; review feedback, docs updated, approval notes (populated during Review)

### WIP Limits
The `wipLimits` object sets maximum card counts per column. The UI warns when a limit is exceeded. Backlog, Ready, and Done have no limits. Current defaults:
- **Design**: 2 — avoid over-planning
- **In Progress**: 3 — focus on finishing before starting
- **Testing**: 3 — keep the test queue short
- **Review**: 5 — human review is the bottleneck, allow a buffer

### Rules for Claude Code
- To move a card: update `column` and `updated` fields only
- To complete a card: set `column` to `"Done"` and update `updated`
- To add a card: append to `cards` array with a new unique `id`
- Before picking up a card: validate the description is sufficient (see Readiness Gate above)
- Before moving a card: check `wipLimits` — do not exceed the limit without flagging it
- To block a card: set `blocked` to `true` and `blockedReason` to the reason
- To unblock a card: set `blocked` to `false` and clear `blockedReason`
- If a card is too large: add breakdown cards to Backlog, note the split in the original card's description
- Update structured sections as work progresses: `requirements` and `design` early, `implementationNotes` during coding, `testPlan` during testing, `reviewNotes` during review
- Only include section fields in JSON when they have content — omit empty sections to keep JSON clean
- Before moving a card to Testing: verify that relevant documentation in `docs/` has been updated
- Never modify the `columns` or `repo` fields
- Always pretty-print JSON with 2-space indentation when writing

## kanban.html — UI Behavior

### Technology
- Plain HTML, CSS, JavaScript — zero dependencies, zero build step
- `fetch()` for reading, `fetch PUT` for writing `kanban.json`
- Served via `serve.js` — a tiny Node.js HTTP server on port 5555 (`node kanban/serve.js`)

### Layout
- Fixed top toolbar with repo name as title
- Horizontal scrolling column layout (one column per kanban stage)
- Cards are vertically stacked within each column
- Responsive card width, columns have a fixed min-width (~280px)

### Card Display
Each card shows:
- Title (bold), with red "bug" badge for defect type
- Priority indicator (colored left border: red = high, yellow = medium, grey = low)
- T-shirt size badge (XS/S/M/L/XL) next to title
- Blocked badge (orange) with reason text when flagged
- Tag pills if present
- Edit and Delete icon buttons on hover

### Interactions
- **Drag and drop** — cards draggable between columns using the HTML Drag and Drop API (no library)
- **Add card** — modal form with fields: title, description, type, priority, size, tags, blocked flag + reason, plus collapsible sections (Requirements, Design, Implementation Notes, Test Plan, Review Notes)
- **Edit card** — same modal pre-populated, triggered by edit icon
- **Delete card** — confirmation prompt, then removes from data
- **Auto-save** — saves to file automatically after every drag, add, edit, or delete
- **Done column** — cards in Done are visually dimmed; a "Clear Done" button appears in that column header

### Visual Design
- Clean, minimal, dark mode aesthetic
- Column headers show card count with WIP limit indicator (turns red when exceeded)
- No external fonts or icon libraries — Unicode characters for icons (✎ ✕ etc.)

## Claude Code Integration

### `/kanban` Skill

The kanban board has a Claude Code skill defined at `.claude/projects/<project>/skills/kanban/SKILL.md`. Invoking `/kanban` bootstraps a session by reading the board state.

**Skill file location** (per-project, not in the submodule):
```
.claude/projects/<project-dir>/skills/kanban/SKILL.md
```

**Skill behavior:**

1. **Session start** — read `kanban/kanban.json`, summarize board state (cards per column, what's in progress), identify highest-priority Ready card, assess readiness, move forward or flag gaps
2. **Session end** — move current card to Testing/Review, add any discovered tasks to Backlog, summarize what was done
3. **Board rules** — the skill references the rules in this spec (WIP limits, column entry policies, structured sections, doc update policy)

**Note:** The SKILL.md lives in the project's `.claude/` directory, not inside the `kanban/` submodule, since skill paths are project-specific. When adding kanban to a new repo, copy the skill template from `kanban/templates/SKILL.md`.

### Skill Roles

| Role | Column | Responsibility |
| --- | --- | --- |
| **Designer** | Design | Reads card, produces implementation plan, populates `design` section |
| **Developer** | In Progress | Picks up card, writes code, populates `implementationNotes` |
| **Tester** | Testing | Writes/runs tests, populates `testPlan`, files defect cards for failures |

### Session Start Pattern
```
Read kanban/kanban.json. Find the highest priority card in the Ready column.
Assess if the description has enough detail. If yes, move it to In Progress and begin work.
If not, update the description with what's missing and leave it in Ready.
```

### Session End Pattern
```
Update kanban/kanban.json: move card [id] to Testing (or Review if tests pass).
Update the appropriate structured sections (implementationNotes, testPlan, etc.).
If any new tasks were discovered during this session, add them to the Backlog.
```

### Claude-Friendly Design Decisions
- JSON is the only data format — no parsing ambiguity
- Column names are explicit strings, not indices
- The `columns` array defines valid column values — Claude can read it to validate moves
- Structured sections (`requirements`, `design`, etc.) keep Claude from clobbering other sections when updating
- Pretty-printed JSON reduces diff noise in git
- All timestamps in ISO 8601

## Git Submodule Usage

The kanban board is designed to be a standalone Git repository that can be added to any project as a submodule. This allows improvements to the UI, server, and spec to be shared across all repositories.

### Repository Structure (as submodule)

```
kanban/                    # ← git submodule
├── kanban.html            # UI — shared across repos
├── kanban-spec.md         # This spec — shared across repos
├── serve.js               # Server — shared across repos
├── CHANGELOG.md           # Schema/structural changes for migration
└── templates/
    ├── kanban.json.template   # Empty board template for new repos
    └── SKILL.md               # Template for the /kanban Claude Code skill
```

Each consuming repo's `kanban.json` is **gitignored in the submodule** but **tracked in the parent repo**. This way:

- The submodule contains only shared code (HTML, JS, spec)
- Each repo has its own board data
- Pulling submodule updates gives you UI/server improvements without touching your cards

### Adding to a New Repo

```bash
# 1. Add the submodule
git submodule add <kanban-repo-url> kanban

# 2. Create your board from the template
cp kanban/templates/kanban.json.template kanban-board.json

# 3. Copy the skill template (adjust project path)
mkdir -p .claude/projects/<project-dir>/skills/kanban
cp kanban/templates/SKILL.md .claude/projects/<project-dir>/skills/kanban/SKILL.md

# 4. Track your board data in the parent repo
git add kanban/kanban.json
```

### Pulling Updates

```bash
cd kanban && git pull origin main && cd ..
git add kanban
git commit -m "Update kanban submodule"
```

If the update includes a schema change (noted in `CHANGELOG.md`), Claude can read the changelog and migrate your `kanban.json` automatically — the changelog includes migration instructions for each breaking change.

### CHANGELOG.md Format

```markdown
## [version] - YYYY-MM-DD

### Schema Changes
- **Added field `foo`** — optional string, no migration needed
- **Renamed field `bar` → `baz`** — migration: rename key in all cards

### Migration (for AI)
For each card in kanban.json:
  - If card has field "bar", rename it to "baz"
  - Add "foo" field only if meaningful content exists (otherwise omit)
```

The "Migration (for AI)" section is written so that Claude Code can read it and apply the changes to any existing `kanban.json` without manual intervention.

## Out of Scope (Keep It Simple)
- No user accounts or auth
- No sync or cloud backup
- No subtasks
- No due dates (priority + column is enough signal)
- No comments (use card description for context)
- No attachments
- No multi-select or bulk actions
